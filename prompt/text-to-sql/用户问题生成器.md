# 问题生成原则

## ✅ 要做
1. **自然表达** - 像真人说话，避免技术术语
   - ✅ "上个月卖了多少钱"
   - ❌ "SELECT SUM(amount) FROM orders"

2. **多样问法** - 同一数据点用不同方式提问
   - "上周销售额是多少"
   - "上周业绩怎么样"
   - "上周营收情况"

3. **覆盖类型** - 包含多种问题类型：
   - **汇总类**："总共/合计/一共多少"
   - **趋势类**："增长/下降/变化趋势"
   - **对比类**："哪个最高/最低/对比"
   - **明细类**："具体有哪些/列表"
   - **占比类**："占比/比例/分布"
   - **条件类**："如果/满足什么条件"

4. **符合口径** - 问题必须在统计口径范围内
   - 如果 SQL 只查"已支付订单"，问题不应包含"未支付"

5. **业务聚焦** - 围绕业务价值提问
   - 如果报表用于监控销售，问题应围绕销售指标

## ❌ 避免
1. 超出 SQL 能力范围的问题
2. 需要额外表才能回答的问题
3. 过于技术化的表述（JOIN、GROUP BY 等）
4. 歧义或模糊的问题
5. 与业务价值无关的边缘问题

# 示例

## 输入示例

```sql
SELECT 
    DATE_FORMAT(order_date, '%Y-%m') AS month,
    SUM(amount) AS total_sales,
    COUNT(DISTINCT user_id) AS buyer_count
FROM orders
WHERE status = 'paid' 
    AND order_date >= DATE_SUB(CURDATE(), INTERVAL 6 MONTH)
GROUP BY month
ORDER BY month;
```

- 报表名称：近 6 个月销售趋势报表
- 业务价值：监控销售走势，及时发现异常波动
- 统计口径：仅统计已支付订单，时间范围为最近 6 个自然月
- 字段描述：
  - order_date: 下单时间
  - amount: 订单金额
  - user_id: 用户 ID
  - status: 订单状态

## 输出示例

### 汇总类
1. 最近 6 个月一共卖了多少钱
2. 这半年有多少用户下单了
3. 近半年的总销售额是多少

### 趋势类
1. 最近几个月销售是增长还是下降
2. 哪个月卖得最好
3. 销售趋势怎么样
4. 这个月比上个月增长了多少

### 对比类
1. 哪个月的销售额最高
2. 最近哪个月表现最差
3. 这个月和去年同期比怎么样

### 明细类
1. 每个月的具体销售额是多少
2. 列出近 6 个月的销售数据
3. 每个月的购买用户数有多少

### 其他类型
1. 平均每个月的销售额是多少
2. 销售数据稳定吗
3. 有没有哪个月异常偏低

### 问题-SQL 映射说明
| 问题示例 | 对应 SQL 逻辑 | 字段映射 |
|----------|---------------|----------|
| 最近 6 个月一共卖了多少钱 | SUM(amount) + WHERE 6 个月范围 | amount→销售额 |
| 哪个月卖得最好 | GROUP BY month + ORDER BY | month→月份 |
| 有多少用户下单 | COUNT(DISTINCT user_id) | user_id→用户 |

### 边界说明
- **可以回答**：近 6 个月、已支付订单、按月统计的销售额和用户数
- **无法回答**：单个订单详情、未支付订单、超过 6 个月的数据、商品维度分析
```

# 注意事项

1. **问题数量**：15-25 个为宜，太少覆盖不足，太多质量下降
2. **语言风格**：根据目标用户群体调整（管理层 vs 一线员工）
3. **迭代优化**：生成后可人工筛选，保留高质量问题
4. **标注意图**：可为每个问题标注 SQL 意图类型（SELECT/AGGREGATE/FILTER 等）
5. **测试验证**：用生成的问题反向测试 Text-to-SQL 模型准确率

# 进阶用法

## 批量生成
如果有多个 SQL/报表，可以：
1. 为每个报表单独生成问题
2. 合并后去重
3. 按业务域分类整理

## 构建测试集
生成的问题可用于：
- Text-to-SQL 模型训练数据
- 准确率测试集
- 用户问法覆盖度分析

## 持续优化
收集真实用户问题，与生成问题对比：
- 哪些问法没覆盖到
- 哪些生成问题用户不会这么问
- 不断调整提示词
```

## 💡 额外建议

### 1. 构建问题-SQL 对数据集
生成问题后，可以整理成训练数据格式：

```json
{
  "question": "最近 6 个月销售额是多少",
  "sql": "SELECT SUM(amount) FROM orders WHERE...",
  "difficulty": "easy",
  "intent": "aggregate"
}
```

### 2. 问题分类标签
为生成的问题添加标签，便于分析：

| 标签 | 说明 | 示例 |
|------|------|------|
| `agg` | 聚合查询 | "总共多少" |
| `trend` | 趋势分析 | "增长还是下降" |
| `compare` | 对比分析 | "哪个最高" |
| `filter` | 条件过滤 | "满足 XX 条件的" |
| `detail` | 明细查询 | "列出所有" |

### 3. 验证生成质量
- 人工审核：业务人员确认问题是否合理
- 反向测试：用生成的问题测试 Text-to-SQL 模型
- 覆盖度分析：对比真实用户问题，检查遗漏
